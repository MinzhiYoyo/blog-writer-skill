# 博客写作技能

撰写高质量技术博客文章的结构化方法。

## 核心原则

### 1. 禁止直接开始写作

不得直接生成博客内容。必须遵循以下流程：

1. 收集用户需求
2. 输出博客写作方案
3. 等待用户确认

只有收到明确批准信号后才开始写作：
- "开始写作"
- "可以开始"
- "开始生成"
- "开始撰写"

### 2. 禁止编造信息

不得编造：
- 数据或统计信息
- 研究论文或引用
- 官方文档或 API
- 配置选项
- 实验结果

若信息无法确认：
1. 向用户提问
2. 查询可靠资料
3. 明确说明无法确认

### 3. 主动发现问题

若发现：
- 信息不足
- 存在歧义
- 描述不完整
- 多个合理解释

必须主动提问。不得猜测或假设。

### 4. 保持一致性

整个博客必须保持：
- 命名一致
- 术语一致
- 风格一致
- 上下文一致
- 逻辑一致

不得出现重复、冲突或前后矛盾。

## 执行流程

严格按以下流程执行，不得跳过步骤。

```
收集需求
    ↓
确认用户输入（包括分部分）
    ↓
生成博客方案及字数估算
    ↓
等待用户确认
    ↓
开始写作
    ↓
自动审查（资深工程师审查，最多5轮）
    ↓
输出最终文件
```

## 用户输入要求

开始写作前必须确认以下内容：

### 输出位置（默认：当前目录）

确认：
- 保存目录
- 文件名称
- 文件格式

默认值：
- 当前目录
- Markdown
- 根据标题自动生成文件名

### 目标读者（默认：技术博客读者）

确认：
- 用户群体
- 前置知识
- 阅读水平

### 写作风格（默认：根据主题自动判断）

支持风格：
- 技术教程
- 原理解析
- 实践经验
- 分享博客
- 科普文章
- 最佳实践

### 文章长度（默认：完整版）

选项：
- 精简版
- 标准版
- 完整版（默认）

### 多部分结构（默认：单部分）

对于复杂或篇幅较长的主题，博客可分成多个部分撰写。

**流程：**
1. 根据主题范围估算总字数
2. 向用户展示字数估算
3. 询问是否需要分成多个部分
4. 若分部分，确认部分数量和命名方式

**常见分部分策略：**
- 按深度：第一部分（基础）、第二部分（进阶）、第三部分（专家）
- 按章节：每个主要章节成为一个独立文件
- 按篇幅：上、中、下三部分

**多部分文件命名：**
- 基础文件名 + 部分后缀：`博客标题-第1部分.md`、`博客标题-第2部分.md`
- 或描述性命名：`博客标题-基础篇.md`、`博客标题-进阶篇.md`

**默认：** 单部分（不分割）

## 内容要求

### 广度

覆盖多个维度，要尽可能保持全面，并且需要向用户确定广度性是否足够
### 深度

每个章节逐层展开，比如：

```
概念
  ↓
原理
  ↓
实现
  ↓
应用
  ↓
实践经验
```

### 客观性

涉及推荐或评价时，说明：
"个人观点，仅供参考。"

## 格式要求

### 标题

- 一级标题：博客标题，章标题
- 二级标题：节标题
- 三级标题：小节标题
- 四级及以上：禁止使用

### 信息密度

优先使用：
- 表格
- 列表
- 引用
- Mermaid 图表
- ASCII 图表

### 列表

复杂内容使用多级列表。

### 表格

若字段名称不能表达完整含义，增加"备注"列。

### 对比

多个方案之间优先使用对比表格。

### 前言

正文开始前，使用一级标题添加前言，包括：
- 面向读者
- 前置知识
- 阅读收益

### 免责声明

在博客标题下面使用粗斜体添加免责声明：`本文部分内容借助 AI 辅助生成，作者已认真审核，如有疏漏欢迎指出，介意者勿阅。`

### 公式

公式和符号第一次出现必须解释。

### 代码

所有代码必须包含：
- 注释
- 参数解释
- 输入输出说明
- 使用场景

## 质量要求

优先保证：
- **正确性**：信息准确
- **完整性**：覆盖必要方面
- **一致性**：无矛盾
- **可读性**：清晰易懂

不得为了缩短篇幅降低内容质量。

## 审查流程

写作完成后，自动进入审查阶段，无需用户确认。

### 审查代理角色

审查由独立的 AI 代理执行，扮演**资深技术写作专家**，具备：
- 10 年以上写作经验
- 博客主题领域的深厚专业知识
- 严格的编辑和质量标准

### 审查标准

审查者评估以下方面：

**内容质量：**
- 技术准确性和正确性
- 覆盖完整性
- 逻辑流畅性和连贯性
- 解释深度

**写作质量：**
- 清晰度和可读性
- 术语一致性
- 语法和风格规范
- 叙事吸引力

**结构与格式：**
- 标题层级规范
- 视觉元素有效性（表格、图表）
- 代码示例质量
- 引用准确性

**专业标准：**
- 读者适配语言
- 观点平衡性与客观性
- 可操作性要点
- 篇幅适当性

### 审查输出

审查者提供：
1. **整体评估**：通过/需修改及摘要
2. **逐节反馈**：各章节具体问题
3. **改进建议**：具体推荐方案
4. **质量评分**：1-10 分及理由

### 审查工作流

```
博客撰写完成
      ↓
自动启动审查代理（资深技术写作专家）
      ↓
审查代理分析内容
      ↓
生成审查报告
      ↓
评估是否需要修改
      ↓
[如需修改] 执行修改 → 重新审查（最多5轮）
      ↓
[无需修改 或 达到5轮] 输出最终文件
```

### 审查迭代规则

- 审查自动执行，无需用户干预
- 每次审查后，如需修改则自动执行修改并重新审查
- 最多执行5轮审查，达到上限后强制输出当前版本
- 审查过程中发现的问题必须记录在审查报告中

### 审查代理调用

启动审查代理时，使用以下提示词结构：

```
你是一名资深技术写作专家，负责审查技术博客文章。

你的角色：
- 作为经验丰富的编辑，具备深厚领域专业知识
- 根据专业技术写作标准评估内容
- 提供建设性、具体的反馈
- 关注清晰度、准确性和读者价值

审查以下博客并提供：
1. 整体评估（通过/需修改）
2. 逐节反馈
3. 具体改进建议
4. 质量评分（1-10）及理由

请全面而建设性地审查。目标是确保博客达到发布标准。
```

## 异常处理

遇到以下情况：
- 信息不足
- 内容冲突
- 无法确认真实性

处理优先级：
1. 向用户提问
2. 联网搜索
3. 明确说明无法确认

不得编造。

## 输出格式

### 单部分博客

生成单个 Markdown 文件，结构如下：

```markdown
# [标题]

***本文部分内容借助 AI 辅助生成，作者已认真审核，如有疏漏欢迎指出，介意者勿阅。***

# 前言

[面向读者、前置知识、阅读收益]

# [章 1]

## [节 1.1]

### [小节 1.1.1]

[内容，使用适当格式：表格、列表、代码块、图表]

# [章 2]

...

# 总结展望

## 本文总结

[关键要点总结]

## 扩展阅读

[参考资料和资源]

# 参考链接

[本文撰写中涉及到的所有链接和引用]
```

### 多部分博客

当博客分成多个部分时，生成独立文件：

**文件结构：**
```
博客标题/
├── 博客标题-第1部分.md
├── 博客标题-第2部分.md
├── 博客标题-第3部分.md
└── README.md（可选：包含各部分链接的索引）
```

**每个部分文件：**
```markdown
# [标题] - 第[N]部分：[部分标题]

# 前言

[面向读者、前置知识、阅读收益]

**"本文部分内容借助 AI 辅助生成，作者已认真审核，如有疏漏欢迎指出。"**

[本部分内容]

## 导航

- 上一篇：[链接到第 N-1 部分]（如适用）
- 下一篇：[链接到第 N+1 部分]（如适用）
- [返回索引](README.md)（如存在）
```

**README.md（可选索引）：**
```markdown
# [标题]

[整个系列的简要描述]

## 各部分

1. [第1部分：标题](博客标题-第1部分.md) - [简要描述]
2. [第2部分：标题](博客标题-第2部分.md) - [简要描述]
3. [第3部分：标题](博客标题-第3部分.md) - [简要描述]
```
